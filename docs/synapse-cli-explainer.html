Synapse v1.6.2

Convex CLI com rota local para dev e produção.

O wrapper synapse convex fica entre o app e o npx convex oficial. Ele descobre qual deployment Synapse usar, injeta as credenciais self-hosted no processo e evita que variáveis da Convex Cloud desviem o comando para o ambiente errado.

v1.6.2 latest dev por padrão deploy vai para prod sem segredo em .synapse/project.json 
$ synapse login https://synapse.exemplo.com
# salva a sessão Synapse fora do projeto

$ synapse select
# escolhe team, project, dev e prod

$ synapse convex dev --once
Using Synapse dev deployment app-dev.

$ synapse convex deploy
Using Synapse prod deployment app-prod.
A ideia fina: o desenvolvedor continua usando a CLI oficial da Convex para o trabalho pesado. O Synapse só resolve o alvo certo e passa CONVEX_SELF_HOSTED_URL + CONVEX_SELF_HOSTED_ADMIN_KEY.

Visão geral

1
Você roda o comando synapse convex dev ou synapse convex deploy dentro do app.
2
O wrapper lê o projeto .synapse/project.json diz qual deployment é dev e qual é prod.
3
Synapse entrega credenciais A API retorna URL e admin key atuais para o deployment escolhido.
4
Ambiente é montado O processo recebe as variáveis self-hosted e perde CONVEX_DEPLOYMENT.
5
CLI oficial executa npx convex ... roda como antes, mas apontado para Synapse.

O que o synapse select deixa pronto

metadata local

.synapse/project.json

Guarda URL do Synapse, team, project e referências sanitizadas dos deployments. É o mapa do projeto, não um cofre.

deployments.devNome/id/status do deployment de desenvolvimento.
deployments.prodNome/id/status do deployment de produção, quando existe.
segredosNão grava admin key nesse arquivo.
ambiente de conveniência

.env.local

Continua sendo atualizado com as credenciais do dev para quem quiser compatibilidade simples. Só que o caminho recomendado agora é synapse convex ..., porque ele reconsulta credenciais frescas no Synapse antes de rodar. 

CONVEX_SELF_HOSTED_URL="..."
CONVEX_SELF_HOSTED_ADMIN_KEY="..."
# CONVEX_DEPLOYMENT fica comentado/removido do caminho crítico

Como dev e produção são escolhidos

development

synapse convex dev

target: dev
comando não é deploy
↓
target inferido como dev
↓
usa deployments.dev.name
↓
roda npx convex dev --once
production

synapse convex deploy

target: prod
primeiro comando é deploy
↓
target inferido como prod
↓
usa deployments.prod.name
↓
roda npx convex deploy
Dá para forçar o destino: synapse convex --target dev deploy manda um deploy para o alvo dev; --target prod faz o inverso quando algum comando normalmente cairia em dev.

Contrato de credenciais

sessão Synapse

synapse login

A sessão fica fora do app, com token de acesso e refresh. Se a API responder 401, o cliente tenta renovar uma vez e repete a chamada.

credenciais frescas

cli_credentials

Para cada execução, o wrapper busca as credenciais do deployment selecionado em vez de depender só do que ficou salvo no arquivo local.

anti Cloud leak

CONVEX_DEPLOYMENT

Quando URL/admin key self-hosted existem, o wrapper remove CONVEX_DEPLOYMENT do ambiente filho para não cair no fluxo da Convex Cloud por engano.

Como instalar em projeto novo

quando estiver no npm

Instalação por app

npm install -D @iann29/synapse cria o bin local. Depois disso, npx synapse ... funciona dentro do app.

release tarball

Antes do npm registry

Instale o asset iann29-synapse-1.6.2.tgz direto da release. O resultado no projeto é igual: npx synapse.

dev local

npm link

Para testar a CLI deste checkout em qualquer pasta: cd cli && npm link. Para um app só, use npm install -D /path/to/convex-synapse/cli e rode npx synapse.

Fix do dashboard incluso na release

antes

Fechar e reabrir perdia o contexto

O modal de upgrade voltava para review a cada abertura. Se o operador fechasse a janela no meio do upgrade, a UI parava de mostrar polling/log/status mesmo com o updater ainda trabalhando.

agora

O modal resume o progresso

Ao iniciar upgrade, ele grava um marcador em sessionStorage. Ao reabrir, consulta /v1/admin/upgrade/status e volta para polling quando o host ainda está em execução.
